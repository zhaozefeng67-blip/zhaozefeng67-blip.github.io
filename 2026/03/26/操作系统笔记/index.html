操作系统笔记

操作系统

操作系统的概念（定义）

操作系统的功能和目标：

  • 操作系统是软件和硬件之间的中间层次，是 系统资源的管理者，既管理了软件资源，也管理了硬件资源。

  • 向上层提供方便易用的服务

  • 是 最接近硬件（裸机）的一层软件

操作系统作为系统资源的管理者

  • 进程/CPU 管理（处理机管理）：Process Management

  • 内存管理（存储器管理）：Memory Management

  • 文件管理：File System Management

  • 设备/IO 管理：I/O Subsystem

  • 操作系统的目标：安全高效（Convenience & Efficiency、Protection & Security）

存储设备排序

registers > cache > Main Memory > nonvolatile disk（SSD，solid state disk, 固态硬盘） > hard-disk drivers (HHD, 机械硬盘) > optical disk（光盘）> magnetic tapes（磁带）

速度越来越慢，存储空间越来越大，价钱越来越便宜

其中前三个是 volatile memory，断电之后，数据直接没了
后面五个是 nonvolatile memory，断电之后，数据还在

向上层提供方便易用的服务

操作系统对外提供更友好的接口（封装思想）

  • 给普通用户用的：

    • 图形化界面（GUI, Graphical User Interface）。

      例如，删除一个文件，在用户哪里只需要拖拽，但是在操作系统可能做了很多

    • 命令接口

      • 联机命令接口

      • 脱机命令接口

  • 给 应用程序 用的

    • 程序接口（即 系统调用）

  • 交互式 命令接口

  • 程序 接口：在程序中进行 系统调用 （System Call ）使用程序接口。普通用户不能直接使用程序接口，只能通过程序代码 间接 使用。

最接近硬件的层次

类比汽车：发动机只会转，轮胎只会滚，可加上一层传动系统，就可以让发动机带着轮子转

那么操作系统也一样，让底层的硬件相互协调配合。

操作系统的运行机制

程序是如何执行的

C 语言代码 -> 编译成机器指令（二进制）

交互式接口：linux 指令，例如 cd, ls

各种程序

  • 内核程序：

    微软、苹果有一帮人负责实现操作系统，他们写的是“内核程序”（Kernel）

    很多内核程序组成了“操作系统内核”，或简称“内核（Kernel）”

    内核 是操作系统最核心的成分，也是 最接近硬件的部分

  • 系统程序：

    操作系统的功能 未必都在内核 中，例如图形化界面 GUI、命令行界面 CLI（Command-Line Interface）

    系统程序（System Programs）主要服务于 用户

  • 中间件：

    中间件（Middleware）主要服务于 应用开发者

  • 应用程序：

    我们普通程序员写的就是“应用程序”（Application Prgrams）

特权指令 vs 非特权指令

操作系统内核作为“管理者”，有时会让 CPU 执行一些“特权指令”，如：内存清零指令。这些指令影响重大，只允许“管理者”——即操作系统内核来使用

在 CPU 设计和生产的时候就划分了 特权指令和非特权指令，因此 CPU 执行一条命令前就能判断出其类型。

内核态 vs 用户态

处于 内核态，说明此时 正在运行的是内核程序，可以执行特权指令

处于 用户态，说明此时 正在运行的是应用程序，此时 只能执行非特权指令

拓展：CPU 中有一个寄存器叫 程序状态字寄存器（PSW），其中有一个二进制位，1 表示“内核态”，0 表示“用户态”

用户态、内核态的切换

内核态 -> 用户态：操作系统内核在让出 CPU 之前，会执行一条特权指令，把 PSW 的标志位设置成“用户态”

用户态 -> 核心态：由 “中断” 引发，硬件自动完成变态过程，触发中断信号意味着操作系统强行夺回 CPU 的使用权。

用户态 -> 核心态：应用程序中出现了一条 特权指令，CPU 发现这是一条特权指令，但又检测到当前处于“用户态”，触发一个 中断信号。CPU 检测到中断信号后，转而运行处理中断信号的内核程序。

中断和异常

中断的作用

中断是 让操作系统内核夺回 CPU 使用权 的唯一途径，使 用户态变成内核态

如果没有中断机制，那么一旦应用程序上 CPU 运行，CPU 就会一直运行这个程序。

中断的类型

  • 内中断：与当前执行的指令 有关，中断信号来自 CPU 内部

    例子 1：试图在用户态执行特权指令

    例子 2：执行除法指令时发现除数为 0

    例子 3：有时候应用程序想 请求系统内核的服务，此时会执行一条 特殊的指令——陷入指令，这个指令会引发一个内中断的信号。执行陷入指令，意味着应用程序将 CPU 控制权还给操作系统内核。“系统调用”就是通过陷入指令完成的。陷入指令是一条特殊的指令，但 不是特权指令

  • 外中断：与当前执行的指令 无关，中断信号来源于 CPU 外部

    例子 1：时钟中断——由时钟部件发出的中断信号

    例子 2：I/O 中断——由输入/输出设备发来的中断信号。

    每一条指令执行结束后，CPU 都会例行检查是否由外中断信号。

大多数教材中、试卷中，“中断”是指狭义的中断（外中断），而内中断一般称为“异常”

中断机制的基本原理

不同的中断信号，需要用不同的中断处理程序来处理。当 CPU 检测到中断信号后，会根据中断信号的类型去查询 “中断向量表”（Interrupt vect），以此来找到相应的 中断处理程序（ISR）在内存中的存放位置。

中断处理步骤：

  1. 保存 CPU 的状态：存储寄存器还有 PC

  2. 确定中断类型：检查中断向量（Interrupt vector），找到对应的 ISR 地址

  3. 在操作系统内核（Kernel）中运行 ISR

系统调用

什么是系统调用

系统调用是操作系统向上 提供给应用程序的接口。

应用程序可以通过 系统调用 的方式请求 操作系统内核的服务。

系统调用和库函数的区别

凡是与共享资源有关的操作（如存储分配、I/O 操作、文件管理），都必须通过系统调用的方式向操作系统内核提出服务请求，由操作系统内核代为完成。这样可以 保证系统的稳定性和安全性，防止用户进行非法操作。

一些 C 语言库函数系统 需要操作系统内核提供的系统调用。

用 C 语言库函数的优点：

可移植性强，源代码在不同操作系统下，只需要重新编译即可。

使用系统调用的优点：

功能更强大，支持创建进程等操作；比库函数快一点

为什么系统调用是必须的

什么功能需要系统调用实现

系统调用的过程

传参数指令，指明系统调用类型，如 fork，以及其他参数。

陷入指令，会引发 CPU 的一个内中断信号。

该中断程序由陷入指令引发，因此转入相应的中断处理程序——即 系统调用的入口程序（属于内核程序，因此在内核态下运行）。

暂停运行应用程序，进入 系统调用入口程序。

系统调用入口程序检查寄存器里的参数，调用与系统调用类型对应的 系统调用的处理程序。例如 fork，那么就会调用 fork 系统调用的处理程序。

CPU 转为用户态，继续执行之前的程序。

陷入指令 是在 用户态 执行的

发出 系统调用请求 是在 用户态，而对 系统调用的相应处理 是在 核心态

操作系统的体系结构

大内核（monolithic kernel, 常考）

  • 优点：性能高，不需要频繁变态

  • 缺点：不够稳定，维护难

微内核（常考）

  • 优点：稳定，容易维护

  • 缺点：性能低，需要频繁变态

分层结构

内核分多层，每一层只能 单向调用（unidirectional） 更第一层 提供的接口。

  • 优点：便于调试和验证（Simplicity of construction and debugging）

  • 缺点：难以定义层（Hard to define layer），性能低（Poor performance），不可跨层调用

模块化

内核 = 主模块 + 可加载内核模块

使用了面向对象的方法（object-oriented approach）

每一个模块都是独立的（seperate），按需加载（is loadable as needed）

可以直接用已知接口和其他进程通信

进程

进程的定义

  • 程序：是静态的（passive），就是个存放在磁盘中的 可执行文件

  • 进程：是动态的（active），是程序的一次执行过程（program in execution in memory）。一个程序（program）可以有多个进程。比如，打开 QQ 三次，就会出现 3 个进程，但 QQ.exe 为一个程序。

  • 进程包含：PCB，程序段、数据段。程序段：程序的代码。数据段：运行过程中产生的各种数据（如程序中定义的变量）

    期中考试回答：进程包括程序段（text section）、数据段（data section）、堆（heap）、栈（stack）

    堆：Containing memory dynamically allocated during run time

    栈：Containing temporary data e.g. local variables

    程序段：the program code

    数据段：Containing global variables and static variables.

  • PCB 是给操作系统用的，程序段、数据段是给进程自己用的。

    PCB 操作系统内核的数据结构（kernel data structure）

    存储在内存里

    对进程不可见（invisible to process itself）

    只能由操作系统内核修改（changed only by）

    每一个进程都有 唯一的 PCB

进程的创建

  • 当创建一个进程时，就创建新的的 PCB；进程结束（dies）后，删除对应的 PCB

  • 当创建好一个进程之后，就会创建一个主线程（Main Thread）。也可以创建多个线程，多个线程共享这个进程的 code（text section）、data（data section）、打开的 file 文件、以及 Heap（动态分配的内存）。但是不共享 registers、stack、PC

  • Multitasking：指的是 多个进程能够“同时”在内存中，并且得到 CPU 的服务

  • 子进程创建的程序：

    • pid_t pid = fork()，虽然只调用了一次，但是会返回两次。

      • 在子进程里面，返回 0

      • 在父进程里面，返回 子进程的 pid

    • wait(&status)，父进程进入阻塞状态，当子进程 exit() 退出之后，即可唤醒父进程

      当唤醒父进程时，子进程的 exit 状态就会写入 status，状态为 0/10/10/1

进程的状态与转换

  • 创建态（new）

  • 就绪态（ready）

  • 运行态（running）

  • 阻塞态（waiting）：比如，当系统请求打印机时，但打印机正在忙，同时又不能一直占用 cpu 资源，所以就进入 waiting 状态。

  • 终止态（Terminated ）。

当进程创建完毕后，进入就绪态，new -> ready

进程被调度，ready -> running

处于阻塞态的进程，它等待的事件发生了（比如打印机），进程从阻塞态回到就绪态。waiting -> ready

时间片到，进程被剥夺 cpu 的使用权，running -> ready

进程用 系统调用 的方式申请某种系统的资源，或者请求等待某个时间发生。

进程 PCB 中，会有一个变量 state 来表示进程的当前状态。

进程的组织——链接方式

操作系统组织 PCB 的方式

  • 链接方式：

    • 执行指针，指向当前处于运行态的进程

    • 就绪队列指针：指向当前处于就绪态的进程，通常会把优先级高的进程放在队头

    • 等待打印机的阻塞队列

    • 等待磁盘的阻塞队列

    很多操作系统还会根据阻塞原因不同，再分为多个阻塞队列

  • 索引方式

进程控制

什么是进程控制

  • 进程控制主要实现进程的状态转换

原语（Primitives）

  • 进程控制使用“原语”实现

  • 原语 是一种特殊的程序，它的 执行具有原子性。也就是说，这段程序的 运行必须一气呵成，不可中断

  • 如何实现原语的原子性：

    “关中断 指令”，执行指令期间，不再检查是否有中断信号

    “开中断 指令”

    关中断指令和开中断指令是 特权指令，只能让 内核程序 使用

  • 原语的类别：

    • 创建原语：操作系统创建一个 进程 用的原语。创建态 -> 就绪态

    • 撤销原语：若该进程正在运行，剥夺其 CPU，终止其所有子进程；将该进程的所有资源归还给父进程或操作系统；删除对应的 PCB

      就绪态/阻塞态/运行态 -> 终止态 -> 无

      引起进程终止的事件：

      • 正常结束（进程自己请求终止）

      • 异常结束（整数除以 0，非法使用特权指令，然后被操作系统强行杀掉）

      • 外界干预（Ctrl + Alt + delete，用户选择杀掉进程）

    • 阻塞原语：保护进程运行现场，将 PCB 状态信息设置为“阻塞态”，暂时停止进程运行；将 PCB 插入对应的事件等待队列里面。运行态 -> 阻塞态

    • 唤醒原语：将 PCB 从 等待队列（waiting） 中删除，设置进程为 就绪态，PCB 插入就绪队列，等待被调度。阻塞态 -> 就绪态

    • 阻塞和唤醒原语总是 成对出现

    • 切换原语：切换进程，一个进程 运行态 -> 就绪态；一个进程 就绪态 -> 运行态

  • 无论哪个进程控制原语，要做的无非三类事情：

    • 更新 PCB 中的信息（修改进程状态，保存/回复运行环境）

    • 将 PCB 插入合适的队列

    • 分配回收资源

  • 程序是如何运行的：

    • 代码 -> 可以执行文件 (*.exe) -> 把程序放入内存 -> CPU 从内存中取出指令

    • CPU 中会设置很多 “寄存器”，用来存放程序运行过程中所需要的某些数据

    • PC：程序计数器，存放下一条指令的地址

    • IR：指令寄存器，存放当前正在执行的指令

    • 通用寄存器：其他一些必要信息

    当执行编译后的命令中，突然切换到了其他进程，那么 PC，以及一些通用寄存器的值需要保存到这个进程的 PCB，因为 PC 和一些通用寄存器并不是某个进程独有的，这些寄存器是属于 CPU 的，并且其他进程也要使用，所以需要保留。

进程通信（IPC）

什么是进程通信

  • 进程通信，是指两个进程之间产生数据交互。

  • 例如，在微博看到了一条吃瓜文，想分享到微信

  • 为什么进程通信需要操作系统支持

  • 进程是分配系统资源的单位（包括内存地址空间），因此 各个进程 拥有的 内存地址空间相互独立。例如，在手机上装了一个垃圾软件，总不能让垃圾软件读取微信的聊天记录等私密的信息吧

共享存储（shared memory）

  • 内存申请一个共享存储区，进程 P 和 Q 可以共享。当进程 P 访问共享存储区时，其他进程（例如进程 Q）不能访问共享存储区。即，为了避免出错，各个进程对共享空间的 访问 应该是互斥的。

  • 可以使用同步互斥工具（如 P、V 操作）

  • 两种共享的方式：

    • 基于数据结构的共享：比如共享空间只能放一个长度为 10 的数组。这种共享方式速度慢、限制多，是一种 低级通信方式

    • 基于存储区的共享：操作系统在内存中划出一块共享存储区，数据的形式、存放位置 都由通信进程控制，而不是操作系统。这种共享方式速度很快，是一种 高级通信方式

消息传递（message passing）

进程间的数据交换，以 格式化的信息 （Message）为单位。进程通过操作系统提供的“发送消息/接收消息”两个 原语 进行数据交换

  • 消息：消息头（发送进程 P 的 ID、接受进程 Q 的 ID） + 消息体

  • 直接通信方式（Direct Communication）：

    消息发送进程要指明接收进程 Q 的 ID

    首先在进程 P 的地址空间中，安装好消息 = 消息头 + 消息体

    然后使用发送原语 send(Q, msg)，把消息从进程 P 的地址空间中，发送到操作系统内核中 Q 的消息队列后。

    然后进程 Q 使用接受原语 receive(P, &msg)，然后操作系统在 Q 的消息队列中检查哪个消息是从 P 发过来的，然后把这个消息 msg 从操作系统内核区复制到接收进程 Q 的内存地址空间。

  • 间接通信方式：

    消息先发到操作系统内核的中间体（信箱）

    进程 P 在进程 P 地址完善消息体 meg 的内容

    进程 P 使用发送原语 send(A , meg)，往 信箱 A 发送消息 meg

    进程 Q 使用接受原语，receive（A, &meg)，从 信箱 A 接受消息

    可以多个进程往同一个信箱 send 消息，也可以从多个进程从一个信箱中 receive 消息

  • 同步（synchronous）和异步（asynchronous）

    • Blocking is considered synchronous，sender 和 receiver 需要等（blocked）直到消息被接收

    • non-blocking is considered asynchronous，sender 和 receiver 不会等，对于 receiver 来说，如果没消息，就接受空消息，直接走人

管道通信

管道是一个特殊的共享文件，又称为 pipe 文件，起始就是在内存中开辟一个 大小固定 的内存缓冲区

共享存储的共享内存区和管道的区别：

  • 共享存储区可以想读哪片区域就读哪片区域，想往哪片区域写，就往哪片区域写

  • 但是管道就像一个 循环 队列，读和写 满足 先进先出。如果前面有空位，P 应该先往前面放位置，读数据只能先读前面放进去的数据。

管道只能采用 半双工通信，某一时间只能实现单向的传输。如果要实现 双向同时通信，则 需要设置两个管道

各个进程要 互斥 地访问管道（由操作系统实现）

当 管道写满 时，写进程 将 阻塞，直到读进程将管道中的数据取走，即可唤醒写进程。

当 管道读取 空，读进程 将 阻塞，直到写进程往管道中写入数据，即可唤醒读进程。

管道中的数据一旦被读出，就彻底消失。因此当多个进程读同一个管道时，可能会错乱。对此，通常有两个解决方案：

  • 一个管道允许多个写进程，一个读进程

  • 允许多个写进程，多个读进程，但系统会让各个读进程轮流从管道中读取数据

Ordinary （anonymous（匿名））pipes：

  • 父进程创建管道和子进程通信

  • 单向的（unidirectional），需要父子关系

  • 必要时，可以创建两个管道

Named pipes：

  • 在没有父子关系的时候，也可以通信

  • 双向的（bidirectional）

线程

什么是线程

在没引入 进程 之前，系统中的各个程序只能串行执行（比如，不能同时聊 QQ + 听音乐）

引入和 进程 之后，可以同时听音乐 + 聊 QQ。

但是 QQ 不止能聊天，可以看视频、文字聊天、传送文件…… 这些功能显然不可能是由一个程序 顺序处理 就能实现的。即有的进程可能需要“同时”做很多事，而传统的进程只能 串行 地执行一系列顺序。为此，引入了“线程”，来增加 并发度

  • 引入线程之后，线程是 调度 的基本单位，进程是 资源分配 的基本单位

  • 引入线程之后，不仅是进程之间可以并发，进程内的各线程之间也可以并发，进一步提升了系统的并发程度

  • 传统的进程间并发，需要切换进程的运行环境，系统开销大。引入线程之后，如果是同一进程内的线程切换，则不需要切换进程环境，系统开销小。

并发与并行

并发（Concurrently）

在单核 CPU 上，通过上下文切换（context switches）处理多个任务（tasks）

并行（Parallesim）

在多核 CPU 上，多个任务（tasks）同时得到执行（simultaneously）

并行意味着（implies）并发，但是并发不意味着并行

  • 数据并行（data parrallelism）：一份数据分成多份，放到不同核心上，执行相同的任务

    例如：有一个长度为 nnn 的数组 aaa 和 bbb，计算 c[i]=a[i]+b[i]c[i] = a[i] + b[i]c[i]=a[i]+b[i]。就可以把数据分成多份，然后放到多个核心上。

  • 任务并行（task parrallelism）：同一份数据，放到不同的核心上，执行不同的任务

    例如：有一个长度为 nnn 的数组 aaa 和 bbb，计算 c[i]=a[i]+b[i]c[i] = a[i] + b[i]c[i]=a[i]+b[i]，d[i]=a[i]∗b[i]d[i] = a[i] * b[i]d[i]=a[i]∗b[i]，e[i]=a[i]/b[i]e[i] = a[i] / b[i]e[i]=a[i]/b[i]。这时候就可以拿出 3 个核心，然后处理三个任务。

线程的实现方式

线程的实现方式

  • 用户级线程（User-Level Thread, ULT）：

    早期操作系统（如：早期 Unix）只支持进程，不支持线程。当时的“线程”是由 线程库 实现的。

    • 线程的管理工作 由谁来完成：用户态的应用程序

    • 线程切换 是否需要 CPU 变态（用户态 -> 内核态）：不需要

    • 操作系统是否能意识到 用户级线程的存在：意识不到

    • 这种线程的实现方式有什么优点和缺点？

      • 优点：在用户空间即可完成，不需要切换到内核态，线程管理的系统开销小，效率高

      • 缺点：如果其中某一个线程被阻塞，整个进程都会被阻塞，并发度不高。多个线程不可以在多核处理机上并发运行（操作系统只能给进程分配一个核心）。

  • 内核级线程（Kernel-Level Thread，KLT，又称“内核支持的线程”）：

    • 线程的管理工作 由谁来完成：操作系统

    • 线程切换 是否需要 CPU 变态（用户态 -> 内核态）：需要

    操作系统是否能意识到 内核级线程的存在：能

  • 这种线程实现方式有什么优点和缺点：

    • 优点：当一个线程被阻塞之后，别的线程还可以继续执行，并发能力强。多线程可以在多核处理机上并发执行

    • 缺点：一个用户 进程 会占用多个内核级线程，线程切换由操作系统内核完成，需要切换到核心态，因此线程管理的成本高，开销大

多线程模型

  • 一对一模型：

    一个用户级线程映射到一个内核级线程。

    • 优点：当一个线程被阻塞后，别的线程还可以继续执行，并发能力强。多线程可以在多核处理机上执行。

    • 缺点：一个用户 进程 会占用多个内核级 线程，线程切换由操作系统内核完成，需要切换到核心态，因此线程管理成本高，开销大。

  • 多对一模型：

    多个用户级线程映射到一个内核级线程

    • 优点：用户级线程切换在用户空间即可完成，无需切换到核心态，管理线程成本低。

    • 缺点：当一个用户级线程被阻塞后，整个 进程 都会被阻塞。多个线程不可以在处理机上 并行运行

    重点重点重点：操作系统只“看得见”内核级线程，因此只有 内核级线程才是处理机分配的单位

  • 多对多模型：

    nnn 个用户级线程映射到 mmm 个内核级线程（n≥mn \geq mn≥m）。

    • 优点：克服了多对一模型 并发度不高 的缺点，又克服了一对一模型中一个用户进程 占用太多内核级线程，开销太大的缺点。

    用户级进程时“代码逻辑”的载体（比如视频聊天的代码逻辑、文字聊天的代码逻辑……）

    内核级线程时“运行机会” 的载体。内核级线程才是处理机分配的单位。

    只有所有内核级线程被阻塞时，整个进程才会被阻塞

调度

调度的三个层次：

  • 高级调度（作业调度）。

    作业：一个具体的任务

    作业调度：按一定的原则，从外存的作业后备队列挑选一个作业调入内存，并创建进程。每个作业只调入一次，调出一次。作业调入时会建立 PCB，调出时才撤销 PCB

  • 低级调度（进程调度）

    低级调度（进程调度/处理机调度） —— 按照某种策略从 就绪队列 中选取一个 进程，将处理机分配给它。

    进程调度是 操作系统最基本的一种调度，在一般的操作系统中都必须配置进程调度。进程调度 的频率很高，一般几十毫秒一次。

  • 中级调度（内存调度）

    内存不足时，可以将某些 进程 的数据调出外存。等内存空间或者进程需要运行时再重新调入内存

    暂时调到外存等待的进程状态为 挂起状态。被挂起的进程 PCB 会被组织成 挂起队列

    中级调度 挂起队列。中级调度 发生的 频率 比 高级调度 高

    有时候手机切换进程的时候，觉得有点卡，一种可能的原因是：那个进程已经被放到了外存

    就绪挂起，阻塞挂起

进程调度（低级调度）

进程调度的时机

进程调度，就是按照某种算法去，从就绪队列中选择一个进程，并为其分配处理机。

  • 什么时候需要进程调度：

    • 当前进程 主动放弃 处理机

      • 程序正常终止

      • 运行过程中发生异常而终止

      • 进程主动请求阻塞（如等待 I/O）

    • 当前进程 被动放弃 处理机

      • 分配给进程的时间片用完

      • 有更紧急的事情要处理（如 I/O 中断）

      • 有更高优先级的进程进入就绪队列

  • 什么时候不能进行进程调度：

    • 在 处理中断的过程中。中断过程复杂，与硬件密切相关，很难做到在中断处理的过程中进行进程切换。

    • 进程在 操作系统内核程序临界区 中。

      临界资源：一个时间段内只允许一个进程使用的资源。各个进程需要 互斥地 访问临界资源

      临界区：访问临界资源的那一段代码

      内核程序临界区（代码） 一般是用来访问 某种内核数据结构 的，比如进程的就绪队列（内核程序临界区）

      内核程序临界区（代码） 访问的 临界资源如果不尽快释放的话，极有可能影响到操作系统内核的其他管理工作。因此在访问内核程临界区期间不能进行调度与切换。

      但是进程在普通临界区中是可以调度、切换的。

    • 在 原子操作过程中（原语）。原子操作不可中断，要一气呵成

进程调度的切换与过程

  • “狭义的调度”与“切换”的区别

  • 进程切换的过程需要做什么

进程调度的方式

  • 非剥夺调度方式（非抢占式）

    只允许进程主动放弃处理机，直到改进程终止或主动要求进入阻塞态。

  • 调度剥夺方式（抢占式）

    当一个进程正在处理机上执行时，如果有一个更重要或更紧迫的进程需要使用处理机，则立即暂停正在执行的进程，将处理机分配给更重要紧迫的那个进程。

    可以优先处理更紧急的进程，也可以实现让各进程按时间片轮流执行的功能（通过时钟中断）。

调度器/调度程序（scheduler）

什么事件会触发“调度程序”

  • 创建新进程

  • 进程退出

  • 运行 进程阻塞

  • I/O 中断 发生（可能唤醒某些阻塞进程）

调度策略：

  • 非抢占式调度策略，只有运行程序阻塞或退出才触发调度程序工作。

  • 抢占式调度策略，每个 时钟中断 或 k 个时钟中断会触发调度程序工作

闲逛进程（idle）

闲逛进程：

  • 优先度最低

  • 0 地址指令（指令周期末尾例行检查中断）

  • 能耗低

调度算法的评价指标

CPU 利用率

早期 CPU 造价极其昂贵，因此人们会希望 CPU 尽可能多地工作

CPU 利用率：值 CPU “忙碌” 的时间占总时间的比例

CPU 利用率 = 忙碌的时间 / 总时间，有的题目还会计算某种设备的利用率

Eg: 某计算机只支持单道程序，某个作业刚开始需要在 CPU 上运行 5 秒，在用打印机打印输出 5 秒，之后再执行 5 秒,才能结束。在此过程中，CPU 利用率、打印机利用率分别是多少？

  • CPU 利用率 = (5 + 5) / (5 + 5 + 5) = 66.66%

  • 打印机利用率 = 5 / 15 = 33.33%

系统吞吐量

对于计算机来说，希望能用尽可能少的时间处理完尽可能多的作业

系统吞吐量：单位事件内完成的作业数量

系统吞吐量 = 总共完成了多少道作业 / 总共花了多少时间

Eg: 某计算机系统处理完 10 道作业，共花费 100 秒，则系统吞吐量为？

  • 10 / 100 = 0.1 道/秒

周转时间

对于计算机的用户来说，他很关心自己的作业从提交到完成花了多长时间

  • 周转时间，是指作业被提交给系统开始，到 作业完成为止 这段时间间隔。对于用户来说，更关心自己单个作业的周转时间。

周转时间 = 作业完成时间 - 作业提交时间

  • 平均周转时间 = 各作业周转时间之和 / 作业数量。对于操作系统来说，更关心系统的整体表现，因此更关心所有作业周转时间的平均值

  • 带权周转时间 = 作业周转时间 / 作业实际运行的时间 = （作业完成时间 - 作业提交时间）/ 作业实际运行的时间

    对于周转时间相同的两个作业，实际运行时间长的作业被服务的时间更多，带权周转时间更小，用户满意度更高

    对于实际运行时间相同的两个作业，周转时间短的带权周转时间更小，用户满意度更高。

    带权周转时间必然 ≥1\geq 1≥1

    带权周转时间与周转时间都是越小越好

  • 平均带权周转时间 = 各个作业带权周转时间之和 / 作业数

等待时间

计算机的用户希望自己的作业尽可能少的等待处理机

等待时间，指进程/作业 处理等待处理机状态时间之和，等待时间越长，用户满意度越低。

作业在后备队列 等待被调度，作业调入内存之后，建立对应的进程。这个进程会被 CPU 服务、会被 I/O 设备服务，当然有时候也会有 等待被服务 的时候

对于 进程 来说，等待的时间就是指进程建立后 等待被服务的时间之和，在等待 I/O 完成的期间其实进程也是在被服务的，所以不计入等待时间

对于 作业 来说，等待时间不仅要考虑 建立进程后的等待时间，还要加上作业在外存后备队列中等待的时间。

等待时间（排队时间） = 周转时间 - 运行时间

响应时间

指的是从用户从 提出请求 到 首次产生响应 所用的时间。

调度算法

各种调度算法的学习思路

  • 算法规则

  • 这种调度算法是用于作业调度 还是 进程调度？

  • 抢占式？非抢占式？

  • 优点和缺点

  • 是否会导致 饥饿（某个进程/作业长期得不到服务）

先来先服务（FCFS）

先来先服务（First Come First Serve）

  • 算法思想：主要是从“公平”的角度考虑（类似我们生活中排队买东西的例子）

  • 算法规则：按照作业/进程到达的先后顺序进行服务

  • 用于作业/进程调度：用于作业调度时，考虑的是哪个 作业 先到达 后备队列；用于进程调度时，考虑的是哪个 进程 先到达 就绪队列

  • 是否可抢占：非抢占式的算法

  • 优缺点：

    • 优点：公平、算法简单

    • 缺点：排在长进程后面的短进程需要等待很长时间，带权周转时间很大，对短作业来说用户体验不好。FCFS 对 长进程有利，对短进程不利（Eg: 排队买奶茶）

  • 是否会导致 饥饿：不会

短作业优先（SJF）

  • 算法思想：追求最少的平均等待时间，最少的平均周转时间，最少的平均带权周转时间

  • 算法规则：最短的作业/进程优先得到服务（所谓的“最短”，是指要求的服务时间最短）

  • 用于作业/进程调度：既可以用于作业调度，也可以用于进程调度。用于进程调度时，称为“短进程调度（SPF, Shortest Process First）优先算法”

  • 是否可以抢占？SJF 和 SPF 时 非抢占式 算法，但是 也有抢占式的版本——最短剩余时间优先 算法（SRTN, Shortest Remaining Time Next）

  • 优缺点：

    • 优点：“最短的” 平均周转时间、平均等待时间

    • 缺点：不公平。对 短作业有利，对长作业不利。可能产生 饥饿 现象

短作业/进程优先调度算法：每次调度选择 当前已经到达 并且 运行时间最短 的作业/进程

对比 FCFS 算法，SPF 算法的平均等待/周转/带权周转时间都要更低

抢占式的短作业优先算法，又称为 最短剩余时间优先算法（SRTN）：

最短剩余时间优先算法，每当有进程加入的时候就绪队列时，就需要调度，如果新到达的进程 剩余时间 比当前运行的进程剩余时间 更短，则由新进程 抢占 处理机，当前运行进程重新回到就绪队列。另外，当一个 进程完成时也需要调度

如果题目 未特别说明，所提到的 “短作业/进程优先算法” 默认 是 非抢占式的

高响应比优先（HRRN, Highest Response Radio Next）

  • 算法思想：要综合考虑 作业/进程 的等待时间和要求服务的时间

  • 在每次调度时计算各个作业/进程的 响应比，选择 响应比最高的作业/进程 为其服务

    响应比 = （等待的时间 + 要求服务的时间（要求服务的时间））/ 要求服务的时间，响应比 ≥1\geq 1≥1

  • 用于作业/进程调度：既可以用于作业调度，也可以用于进程调度

  • 是否可以抢占：非抢占式 的算法，因此只有当当前 作业/进程 主动放弃处理机的时候，才需要调度，才需要计算响应比

  • 优缺点：

    • 优点：综合考虑了等待时间和运行时间（要求服务的时间）

    • 缺点：基于高频率的进程切换，有一定开销；不区分任务的紧急程度

    • 是否会导致饥饿：不会

  • 是否会导致饥饿：不会

时间片轮转算法（RR, Round-Robin）

  • 算法思想：公平、轮流地为各个进程服务，让每一个进程在一定时间间隔都可以得到响应

  • 算法规则：按照各个进程到达就绪队列的时间，轮流让各个进程执行一个 时间片（如 100ms）。若进程未在一个时间片内执行完，则剥夺处理机，将进程重新放回就绪队列重新排队。

  • 用于作业/进程调度：用于 进程调度（只有作业放入内存建立了相应的进程后，才能被分配处理机时间片）

  • 是否可抢占：抢占式，由时钟装置发出 时钟中断 来通知 CPU 时间片已到

  • 优缺点：

  • 是否会导致饥饿：

如果 时间片 太大，时间片轮转算法 就会退化成 先来先服务算法，并且会增大 进程响应时间。因此时间片不能太大。

另一方面，如果时间片太小，会导致进程切换过于频繁。

优先级调度算法

  • 算法思想

  • 算法规则：每一个进程/作业有各自的优先级，调度时选择优先级最高的作业/进程

  • 用于作业/进程调度：既可以用于作业调度，也可以用于进程调度，甚至可以用于之后需要学习的 I/O 调度

  • 是否可以抢占：有抢占式的，也有非抢占式的。

  • 优缺点

  • 是否导致饥饿

优先级可以分为 静态优先级 和 动态优先级

系统进程优先级 高于 用户进程

前台进程优先级 高于 后台进程

操作系统更 **偏好 I/O 型进程（或称 I/O 繁忙型进程）

与 I/O 型进程 相对的是 计算型进程（或称 CPU 繁忙型进程）

I/O 设备和 CPU 可以 并行工作，优先让 I/O 繁忙型进程优先运行的话，资源的利用率，系统的吞吐量都会上升。

多级反馈队列调度算法

  • 算法思想：对其他调度算法的这种权衡

  • 算法规则：

  • 用于作业/进程调度：用于 进程调度

  • 是否可以抢占？抢占式 算法

  • 是否会导致饥饿：会

设置多级 就绪队列，各级队列 优先级 从 高到低，时间片 从 小到大

新进程 到达时，先进入 1 级队列，按照 FCFS 原则 排队等待被分配时间片。若用完时间片进程还 未结束，则进程 进入下一级 队列队尾。如果此时已经在最下级的队列，则 重新放回 最下级队列队尾。
